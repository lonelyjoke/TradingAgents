# TradingAgents A股投研系统当前功能总览

本文记录当前 A股投研系统的真实能力边界，重点说明：已经接入哪些研究工具包，系统如何从数据获取走到分析师辩论和 PM 报告，以及最终报告应包含哪些内容。

## 1. 当前定位

当前系统不是自动交易程序，也不是保证收益的选股器。它更接近一个 A股买方投研工作台：

1. 先做数据预检，确认行情、估值、三张表、现金流和可读财报正文可用。
2. 再预取通用研究上下文和 gated 行业工具包上下文。
3. 让市场、新闻、基本面分析师分别形成初步判断。
4. 让多头、空头和研究经理围绕同一组证据辩论。
5. 让交易员、风险角色和组合经理把结论收敛成 PM 报告、评级、仓位动作和验证清单。

核心原则是：先保证证据链，再给结论；先解释公司怎么赚钱，再讨论估值和评级。

## 2. 数据硬门槛

A股标的在进入 LLM 生成前会先跑 preflight。当前硬门槛包括：

- `stock_basic`：公司名称、行业、上市地等基础画像。
- `daily`：最新行情数据，不能明显滞后。
- `daily_basic`：估值、市值、换手率等交易快照。
- `fina_indicator`：ROE、ROA、毛利率、净利率、成长性等财务质量指标。
- `income`：利润表关键字段。
- `balancesheet`：资产负债表关键字段。
- `cashflow`：现金流量表关键字段。
- `filing_text`：可读年报、半年报或季报正文。

如果第一层结构化数据或第二层可读财报正文失败，系统会在 LLM 生成前停止，不再生成证据不足的报告。

财报正文链路现在包括 Tushare 公告、巨潮 CNINFO fallback、PDF 下载、`pdftotext` 解析、本地 disclosure cache 恢复和标题过滤。提示性公告、预约披露时间公告等不会被当成正式财报正文。

## 3. 通用研究层

当前系统会为 A股预取 15 个通用研究上下文：

| 模块 | 作用 |
| --- | --- |
| `thematic_catalyst` | 从公告、财报、新闻、互动等线索识别题材和催化剂，并区分硬催化、软催化和叙事期权。 |
| `commodity_product_price` | 为周期、资源、原材料相关公司补充商品价格或产品价格线索。 |
| `price_move_attribution` | 拆解股价近期变化与指数、行业、事件和基本面的关系。 |
| `shipping_cycle` | 为航运/港口/海运链标的补充运价、航线和周期变量。 |
| `financial_report_intelligence` | 深读财报正文，提取业务模式、分部经济性、管理层讨论、会计质量、现金流和附注线索。 |
| `peer_comparison` | 在同业中比较估值、盈利、成长、负债、股息和综合打分。 |
| `supply_chain_comparison` | 判断公司在产业链中的位置，并比较上下游和替代表达。 |
| `earnings_model` | 用最近报告期、年化、季节性调整和利润率/现金流指标形成盈利基准。 |
| `market_expectation` | 用市值、PE/PS/PB 和利润基准倒推市场隐含预期。 |
| `price_eps_pe_decomposition` | 拆解股价变化来自 EPS、估值倍数还是二者共同作用。 |
| `management_capital_allocation` | 检查分红、回购、融资、投资、减持、资本开支和管理层资本配置。 |
| `shareholder_structure` | 分析股东结构、减持、质押、集中度、限售和潜在供给压力。 |
| `investor_interaction` | 读取交易所互动平台问答，区分公司正式回应、商业秘密回避和可验证线索。 |
| `policy_planning` | 接入政策、规划和行业文件，区分国家方向、行业需求斜率和公司受益。 |
| `web_fact_check` | 对高频但 Tushare/财报覆盖不足的事实做外部网页交叉核验。 |

这些模块是基础研究底座，不等同于行业工具包。

## 4. 行业和赛道工具包

当前已接入 11 个 gated 行业/赛道工具包。它们只有在公司名称、行业、财报正文、公告或互动平台证据触发时才进入分析，不会强行套用到无关公司。

| 工具包 | 适用对象 | 核心变量 |
| --- | --- | --- |
| 白酒 | 白酒、酒类品牌 | 批价、渠道库存、合同负债、回款、节日需求、产品结构、同业酒企对比。 |
| 算力租赁 | GPU/IDC/AI 算力租赁 | GPU 型号和数量、客户合同、租金、利用率、电力/机柜/带宽/O&M、折旧和 ROIC。 |
| 高股息防御 | 高分红、类红利资产 | 股息覆盖、利润稳定性、现金流、资本约束、估值缓冲和股息陷阱。 |
| 建材 | 水泥、玻璃、玻纤、防水、管材、涂料、瓷砖等 | 地产/基建/翻新需求、价格、库存、产能利用率、原燃料成本、应收和现金回款。 |
| 大消费/食品饮料 | 食品、饮料、乳品、肉制品、调味品、速冻、预制菜、休闲食品 | 餐饮/家庭需求、渠道动销、经销商库存、原材料、促销、冷链物流、食品安全、品类结构。 |
| 光模块/AI 数据通信 | 中际旭创、新易盛、天孚通信等光通信链 | 800G/1.6T、AI capex、客户认证、产能/良率、ASP、库存、应收、OCF、CPO/LPO/硅光路线。 |
| 创新药/医药服务 | 创新药、Biotech、CRO/CDMO、药企 | 管线、临床阶段、注册审批、医保/集采、商业化销售、现金 runway、订单和产能利用率。 |
| 软件/SaaS | 软件、金融 IT、工业软件、网络安全、AI 软件 | ARR/ARPU、付费用户、续费/流失、合同负债转化、项目验收、回款、AI 变现和算力成本。 |
| 保险 | 寿险、财险、综合金融保险集团 | NBV、EV、偿付能力、代理人/银保、投资收益率、COR、分红和资本约束。 |
| 医疗器械 | 医疗设备、IVD、耗材、高值器械 | 装机量、试剂拉动、招采/VBP、注册证、海外渠道、服务收入、应收和现金转化。 |
| 金属矿业 | 黄金、铜、铝、锂、锌、镍、银、采矿/冶炼 | 储量、品位、权益产量、AISC/单位成本、项目爬坡、金属价格、套保、NAV/SOTP。 |

另外，`commodity_product_price` 和 `shipping_cycle` 虽然是通用上下文，但在周期、资源、航运类公司中也承担行业景气工具的作用。

## 5. 分析链路

当前链路可以理解为六层递进：

1. 数据层：Tushare、CNINFO、本地 cache、网页事实核验、互动平台、政策文件。
2. 预检层：确认第一层结构化数据和第二层财报正文可读。
3. 上下文层：预取通用研究模块和行业工具包，并生成 `data_coverage` 审计。
4. 分析师层：市场、新闻、基本面分析师分别阅读上下文，形成初始报告。
5. 辩论层：多头提出可下注理由，空头攻击证据缺口和估值过度，研究经理收敛分歧。
6. 决策层：交易员提出动作，风险角色审查仓位和下行情景，组合经理输出最终 PM 报告。

`data_coverage` 会把每个模块标成 `ready`、`partial`、`failed`、`missing` 或 `not_applicable`。PM 不允许把失败模块当成中性证据；如果缺口触及核心投资假设，必须降低确信度或停止生成。

## 6. PM 报告应包含的内容

当前 PM 报告应从“堆材料”转向“层层递进”：

1. 最终评级、动作和时间窗口。
2. 一句话投资结论：为什么现在是 Buy、Overweight、Hold、Underweight 或 Sell。
3. 数据覆盖审计：哪些核心证据 ready，哪些 partial/failed。
4. 公司商业模式和产业链位置：公司靠什么赚钱，处于上游/中游/下游哪个环节，上下游有哪些上市公司或可比标的。
5. 财报和业务拆解：收入、利润、现金流、资产负债表、分部业务、第二增长曲线和会计质量。
6. 行业工具包观点：如果触发行业工具包，必须使用该行业的原生变量，而不是泛泛讨论估值和增长。
7. 市场预期和估值：市值隐含利润、PE/PB/PS 分位、EPS/估值倍数拆解、情景估值。
8. 同业和替代表达：为什么买它，而不是买更便宜、更高质量或更直接的同行。
9. 多空分歧：多头最强证据、空头最强反驳、研究经理如何裁决。
10. 风险、验证清单和降级/升级触发器：下一期财报、公告、行业数据或价格行为如何证伪判断。

## 7. 当前边界

- 行业工具包已经覆盖多个核心赛道，但仍不是所有行业都有专用框架。
- 部分高频行业数据仍依赖网页核验或人工补充，例如渠道库存、终端动销、真实 ASP、客户订单和海外云厂商采购节奏。
- 财报 PDF 文本解析已经加硬门槛，但外部公告源、网络和 PDF 格式仍可能偶发失败；失败时系统应停在数据层，而不是生成浅报告。
- 同业比较仍以 Tushare 行业标签和可获得财务数据为基础，不能完全替代人工定义真实竞品。
- PM 报告的研究深度取决于数据源是否完整、行业工具包是否触发、以及分析链路是否把证据转化为投资判断。

## 8. 调试原则

以后修复数据或工具包问题时，优先采用“小测试脚本先验收，再跑完整报告”的方式：

1. 先单测失败的数据函数或上下文函数。
2. 确认返回 shape、字段、正文字符数或 audit 状态。
3. 再跑完整 CLI 报告。

这样可以避免每次用完整研报来测试底层数据链路，提高迭代效率。
