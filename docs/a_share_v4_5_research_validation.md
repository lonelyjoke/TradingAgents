# TradingAgents v4.5 投研观点验证系统

v4.5 的目标不是立刻做一个复杂量化交易平台，而是先回答一个更基础的问题：

```text
我们的投研系统给出的观点，到底有没有预测能力？
```

## 1. 为什么需要验证系统

v4.4 之后，系统加入了很多投研框架：

- 核心下注主线；
- 预期差；
- 胜率赔率；
- 景气与估值校准；
- 研究缺口；
- 供需替代分析；
- 止盈和进场观察区间。

这些规则听起来合理，但如果没有验证，仍然可能变成主观 prompt。

所以 v4.5 先建立一套“观点事后验证系统”，用真实行情去检验报告结论。

## 2. 第一版验证什么

第一版不验证复杂交易策略，只验证报告方向。

系统会读取保存下来的报告目录，例如：

```text
reports/601872.SH_20260511_203213/
```

然后解析：

- 标的代码；
- 报告生成日期；
- 最终评级；
- 核心下注主线；
- 观点强度与仓位。

之后用 Tushare 获取真实后续价格，计算：

- 未来 20 个交易日收益；
- 未来 60 个交易日收益；
- 未来 120 个交易日收益；
- 相对沪深300收益；
- 最大回撤；
- 最大浮盈；
- 观点是否命中。

## 3. 怎么判断命中

第一版采用简单规则：

- `Buy` / `Overweight`：如果后续跑赢沪深300，算命中；
- `Underweight` / `Sell`：如果后续跑输沪深300，算命中；
- `Hold`：如果后续相对沪深300波动不大，默认正负 2% 以内，算命中。

这不是完美规则，但足够帮助我们发现系统偏差：

- 是否总是过度保守；
- 是否 Buy/Overweight 没有跑赢能力；
- 是否 Underweight/Sell 真的能规避风险；
- 是否 Hold 只是缺乏观点。

## 4. 怎么运行

在项目根目录运行：

```powershell
.\.venv\Scripts\python.exe scripts\validate_research_reports.py --reports reports --horizons 20 60 120
```

输出两个文件：

```text
reports_validation.csv
reports_validation_summary.csv
```

其中：

- `reports_validation.csv` 是每一份报告、每一个周期的详细结果；
- `reports_validation_summary.csv` 是按评级和周期汇总的命中率、平均收益、平均超额收益、平均回撤。

如果只想验证某一份报告：

```powershell
.\.venv\Scripts\python.exe scripts\validate_research_reports.py --reports reports\601872.SH_20260511_203213 --horizons 20 60 120
```

## 5. 当前版本的边界

第一版验证系统有几个重要边界：

- 它只使用 Tushare 真实行情；
- 如果没有足够未来交易日，会显示 `insufficient_future_data`；
- 如果 Tushare 拉不到价格，会显示 `price_unavailable`；
- 不会用样本数据或模拟数据替代真实价格；
- 入场价格默认用报告生成后下一个交易日的收盘价，避免使用报告已经看到的同日收盘。

## 6. 为什么不一开始做完整回测

完整组合回测当然重要，但第一步不应该太复杂。

原因是：

- 我们首先要知道单份报告的方向有没有用；
- 如果报告方向本身没有预测力，组合回测只会把问题变复杂；
- 当前系统的输出是投研观点，不是机械交易信号；
- 先做观点验证，可以更快发现 prompt 和框架偏差。

更合理的路线是：

```text
第一步：观点验证
第二步：分评级统计
第三步：分行业/分估值/分景气环境统计
第四步：影子组合回测
第五步：正式策略回测
```

## 7. 如何缩短迭代周期

如果只等 20/60/120 个交易日，迭代会太慢。

所以建议并行做三类验证：

### 真实向前验证

从今天开始，保存每次真实生成的报告，未来定期验证。

这是最可靠的验证，但周期较长。

### 历史案例复盘

选过去典型案例，用当时之前的数据生成报告，再看后续表现。

注意：这类验证要尽量避免未来信息泄漏。

### 黄金样本测试集

人工建立一批典型案例：

- 低估值低景气但后来修复；
- 低估值但价值陷阱；
- 高景气高估值且后来回落；
- 高景气但仍有预期差；
- 红利稳健股；
- 周期底部反转股。

每次迭代后，看系统是否能给出符合投资常识的判断。

## 8. 怎么使用验证结果迭代系统

验证结果不是为了追求每次都对，而是找系统性偏差。

例如：

- 如果低估值低景气标的长期被 Underweight，但后续经常跑赢，说明系统过度惩罚低景气；
- 如果高景气高估值标的经常被 Overweight，但后续回撤大，说明系统过度奖励景气；
- 如果 Hold 后续经常大涨或大跌，说明 Hold 被滥用；
- 如果 Overweight 不能产生正超额收益，说明预期差判断不够好；
- 如果 Underweight 不能规避下跌，说明风险识别不够好。

最终目标不是让系统永远给乐观评级，而是让它在不同市场环境下更公允、更可检验。
